# UBID Platform — File Structure Reference Guide
### What Every File Does & Why It Exists

---
## Root Level

| File | What It Does |
|---|---|
| `docker-compose.yml` | Defines and starts the entire stack (PostgreSQL, Nominatim, MLflow, Backend, Frontend) with one command: `docker compose up` |
| `.env` | Stores all secrets and config values (DB password, scrambler key, API URLs). Never committed to Git. |
| `.env.example` | A safe template of `.env` with placeholder values. Committed to Git so teammates know what variables are needed. |
| `README.md` | Project overview, setup instructions, and how to run the demo. |

## `data/` — Generated Synthetic Data *(git-ignored)*

This entire folder is generated by scripts and never committed. It simulates what real department system exports would look like.

### `data/raw/` — Simulated Department Exports

| File | What It Does |
|---|---|
| `shop_establishment.csv` | Simulated export from the Shop Establishment department. Contains ~8,000–10,000 rows with business names, addresses, owner details, trade categories, registration dates. Names and addresses are intentionally messy — abbreviations, different formats, missing PANs. |
| `factories.csv` | Simulated export from the Factories department. Contains factory licence numbers, NIC codes, worker counts, survey-number-style addresses. ~35% of rows have a PAN field populated. |
| `labour.csv` | Simulated export from the Labour department. Contains employer codes, mixed Kannada/English addresses, industry type as free text. |
| `kspcb.csv` | Simulated export from Karnataka State Pollution Control Board. Consent order numbers, NIC codes, consent validity dates. Highest PAN/GSTIN coverage (~65%) among all departments. |
| `activity_events.csv` | 12 months of activity events across all businesses — licence renewals, inspections, electricity consumption readings, compliance filings, closure declarations. ~120,000 rows. Each row links to a source system record. |

### `data/processed/`
Stores the normalised versions of raw records after the Normalisation Engine (Phase 3) runs. Each record has canonical name, parsed address components, validated PAN/GSTIN, geocoordinates, and phonetic keys. Used as input to the Entity Resolution pipeline.

### `data/ground_truth/`

| File | What It Does |
|---|---|
| `labelled_pairs.csv` | Training and evaluation data for the ML model. Each row: `(pair_id, record_a_id, record_b_id, label)` where label=1 means same business, label=0 means different. ~10,000 rows. Generated from known entity clusters. |
| `entity_clusters.csv` | The ground truth mapping: `(entity_id, source_system, source_record_id)`. Tells you which department records belong to the same real-world business. Used to generate labelled pairs and to evaluate classification accuracy after the pipeline runs. Never fed into the pipeline itself. |

## `scripts/` — One-Shot Execution Scripts

These are run directly from the command line, not imported as modules. They orchestrate the full pipeline.

| File | What It Does |
|---|---|
| `generate_synthetic_data.py` | **Run once at the start.** Creates all CSVs in `data/raw/` and `data/ground_truth/`. Generates 5,000 ground truth entities, produces 15,000–20,000 department records with injected variations, and generates 12 months of activity events. Takes ~2–3 minutes. |
| `run_pipeline.py` | **Run before every demo.** Executes the full pipeline end-to-end: loads source records → normalises → blocks → extracts features → scores → assigns UBIDs → routes activity events → computes activity scores. Populates all UBID registry tables. Takes ~10–15 minutes. |
| `train_model.py` | **Run once after data generation.** Trains the LightGBM classifier on labelled pairs, applies Platt Scaling calibration, saves `lgbm_model.pkl` and `calibrated_model.pkl` to `src/entity_resolution/models/`. Also logs the experiment to MLflow. |
| `reset_demo.py` | **Run before demo rehearsal.** Wipes all UBID registry tables (entities, source links, evidence, review tasks, activity scores) but leaves source data intact. Run this then immediately run `run_pipeline.py` to get a fresh, reproducible demo state. |

## `src/` — All Application Source Code

---

### `src/data_generation/` — Synthetic Data Generators

These files only run during data generation. They are never called by the live API.

| File | What It Does |
|---|---|
| `entity_generator.py` | Creates the 5,000 ground truth business entities. Each entity has a true name, true PAN/GSTIN, true address, NIC code, owner name, activity status (active/dormant/closed), and whether it is seasonal. This is the single source of truth that all department records are derived from. |
| `department_record_generator.py` | Takes each ground truth entity and generates 2–5 department records for it across the 4 departments. Decides which departments a business appears in based on business type (factory vs. shop vs. service). Each record gets the department's primary ID format (SE Reg No, Factory Licence No, etc.). |
| `variation_injector.py` | The most important generation file. Takes clean department records and injects realistic messiness: legal suffix variations (Pvt Ltd vs P Ltd vs Private Limited), abbreviations (Inds vs Industries), typos, Kannada/English transliteration differences, address format switching, PAN absent/corrupted. Also injects 8% intra-department duplicates per department. |
| `activity_event_generator.py` | Generates 12 months of activity events per entity. Decides event frequency based on ground truth status (active businesses get monthly electricity readings + annual renewals + 1–3 inspections; dormant get sparse signals; closed get a closure declaration event). Applies seasonal patterns for NIC codes that have known seasonal cycles. |

#### `src/data_generation/dictionaries/`

| File | What It Does |
|---|---|
| `karnataka_business_names.py` | A pool of ~2,000 realistic Karnataka-style synthetic business names used by the PII scrambler. Names follow Karnataka linguistic patterns (Kannada-origin names, common industry suffixes). Ensures scrambled names look plausible, not random strings. |
| `karnataka_street_names.py` | Pool of synthetic street names for address scrambling. Includes BBMP-style names (3rd Cross, 5th Main), industrial area names (KIADB, KSSIDC plots), and rural locality names. |
| `nic_codes.py` | Full NIC 2-digit and 4-digit code list with descriptions and industry sector labels. Used by entity generator to assign realistic industry codes and by feature extractor for NIC compatibility scoring. |
| `pin_codes.py` | Karnataka pin code list with district, taluk, and locality mappings. Also contains the pin adjacency table (which pins are geographically neighbouring) used in Feature F06. Centroids (lat/lng) for each pin code used as geocoding fallback. |

---

### `src/normalisation/` — Normalisation Engine

These files transform raw, messy department records into clean, comparable internal records. Every record passes through these before any ML stage.

| File | What It Does |
|---|---|
| `name_normaliser.py` | Takes a raw business name and produces: (1) a canonical comparison string (stripped of legal suffixes, abbreviations expanded, transliterated, punctuation removed), (2) a Soundex key for phonetic blocking, (3) a Double-Metaphone key for phonetic blocking. The original name is always preserved — canonical form is used only for matching. |
| `address_parser.py` | Parses free-text Karnataka addresses into structured components: building number, street, locality, landmark, ward, taluk, district, pin code, industrial area name, survey/plot number. Handles all 5 Karnataka address format types: BBMP ward style, KIADB industrial estate style, rural survey number style, landmark-based, and minimal (locality + pin only). Also generates a set of address tokens for Jaccard similarity matching. |
| `identifier_validator.py` | Validates and normalises PAN (5-alpha + 4-numeric + 1-alpha format) and GSTIN (15-char, state code must be 29 for Karnataka). Returns whether the value is present, valid, and normalised. Handles common null representations (NA, NIL, N/A, NULL, empty string). |
| `pii_scrambler.py` | Applies deterministic scrambling to all PII fields before they reach any ML stage. Uses HMAC-SHA256 so the same input always produces the same output (reproducible test runs). Scrambles business names (via synthetic name dictionary lookup), PAN/GSTIN (structure-preserving so format validity is maintained), phone numbers (digit substitution cipher), and dates (fixed offset shift). |
| `geocoder.py` | Calls the self-hosted OSM Nominatim instance to convert parsed addresses into lat/lng coordinates. Returns a quality flag: HIGH (street-level match), MEDIUM (locality-level), LOW (pin code centroid fallback). Geo-distance feature (F07) is only computed when both records have quality ≥ MEDIUM. |
| `standardiser.py` | Orchestrator that calls the above five modules in the correct order for each raw record. Input: one raw department record dict. Output: one standardised record in the unified internal schema with all normalised fields, phonetic keys, geocoordinates, and scrambled PII. This is the only file the pipeline needs to call for normalisation. |

### `src/entity_resolution/` — Entity Resolution ML Pipeline

The technical core. Takes normalised records and determines which ones refer to the same real-world business.

| File | What It Does |
|---|---|
| `blocker.py` | Reduces the O(n²) comparison problem to a manageable candidate pair set. Groups records by 6 different blocking keys (PAN exact, GSTIN exact, Pin+Soundex, Pin+Metaphone, H3 geo-cell+name token, NIC+pin+name token). A pair is generated if they share any blocking key. Target: ≥99.5% blocking recall — no more than 0.5% of true matching pairs should be missed here. |
| `feature_extractor.py` | Computes the 13-feature vector for each candidate pair. Features cover: name similarity (F01 Jaro-Winkler, F02 token set ratio, F03 abbreviation match), identifier matching (F04 PAN, F05 GSTIN — with −1 for confirmed mismatch), location (F06 pin code, F07 geo distance, F08 address token overlap), contact (F09 phone), industry (F10 NIC compatibility), person (F11 owner name), and structural flags (F12 same-source, F13 registration date proximity). |
| `scorer.py` | Loads the trained LightGBM model, runs inference on a feature vector, applies Platt Scaling calibration to get a properly calibrated confidence score, routes the pair to Auto-Link / Review Queue / Keep Separate based on thresholds, and runs SHAP TreeExplainer to produce per-feature contribution values. Also contains the `train_model()` function used by `scripts/train_model.py`. |
| `ubid_assigner.py` | Runs Union-Find (Disjoint Set Union) over all Auto-Link pairs to form entity clusters via transitive closure. Each connected component gets one UBID (`KA-UBID-XXXXXX` in base-36). Anchors the UBID to PAN/GSTIN if present in any cluster member. Writes all UBID entities, source links, and evidence records to PostgreSQL. |

#### `src/entity_resolution/models/`

| File | What It Does |
|---|---|
| `lgbm_model.pkl` | The raw trained LightGBM classifier. Used by the SHAP TreeExplainer for exact Shapley value computation. Not used directly for scoring (use the calibrated model for that). |
| `calibrated_model.pkl` | The Platt-Scaled calibrated wrapper around the LightGBM model. This is what produces the final confidence score. A score of 0.85 from this model means 85% of such pairs are truly matches in validation data — not just a relative rank. |

### `src/activity_engine/` — Activity Intelligence Engine

Processes the event stream and classifies each UBID as Active, Dormant, or Closed.

| File | What It Does |
|---|---|
| `signal_config.py` | Central configuration file for all signal types. Defines the base weight (+0.90 for high electricity consumption, −1.00 for closure declaration, etc.) and half-life in days for each event type. Changing these values tunes the classifier's sensitivity. Also defines the NIC codes that have known seasonal patterns and the Active/Dormant/Closed score thresholds. |
| `event_router.py` | Polls unprocessed events from the `activity_events_raw` table. For each event, looks up the source record ID in `ubid_source_links` to find the UBID. If found, attaches the UBID and writes to `ubid_activity_events`. If not found, routes to `unmatched_events` with a triage reason code. Events are never silently dropped. |
| `signal_scorer.py` | For each UBID, fetches all events in the 12-month lookback window and computes the Activity Score using the time-decay formula: `AS = Σ [weight × e^(−λ × days_since)]`. Checks for hard permanent signals (closure declaration, licence cancellation) first — these override all other signals regardless of score. Normalises the raw score to [−1, +1] via sigmoid transform and returns a full evidence list showing each event's contribution. |
| `activity_classifier.py` | Applies the final classification thresholds to the Activity Score: AS > 0.4 → Active, −0.2 to 0.4 → Dormant, < −0.2 → Closed Suspected, hard closure event → Closed Confirmed. Writes the result plus the full evidence snapshot as JSONB to the `activity_scores` table. Also handles seasonal business adjustments using industry-specific scoring profiles. |

### `src/database/` — Database Layer

| File | What It Does |
|---|---|
| `connection.py` | Creates the SQLAlchemy engine from environment variables, defines the `SessionLocal` factory, and provides the `get_db()` dependency function used by all FastAPI route handlers for session injection. |
| `models.py` | Defines all 11 SQLAlchemy ORM table classes: 5 source system tables (dept_shop_establishment, dept_factories, dept_labour, dept_kspcb, activity_events_raw) and 6 UBID registry tables (ubid_entities, ubid_source_links, ubid_link_evidence, review_tasks, ubid_activity_events, activity_scores, unmatched_events). Includes all indexes for query performance. |
| `migrations/` | Alembic-managed folder. Contains auto-generated migration scripts that version-control the database schema. Run `alembic upgrade head` to apply all migrations. Run `alembic revision --autogenerate` when you change `models.py`. |

### `src/api/` — FastAPI Backend

| File | What It Does |
|---|---|
| `main.py` | FastAPI application entry point. Creates the app instance, registers CORS middleware (so the React frontend can call the API), and mounts all 4 routers under their URL prefixes. Also defines the `/health` endpoint. |
| `dependencies.py` | Shared FastAPI dependencies injected into route handlers. Contains `get_db()` for database sessions and auth stub functions (for demo: no real auth; for production: JWT validation against Karnataka government IdP). |

#### `src/api/routers/`

| File | What It Does |
|---|---|
| `ubid.py` | Handles all UBID lookup and detail requests. Endpoints: `GET /api/ubid/lookup` (search by PAN, GSTIN, or name+pin), `GET /api/ubid/{ubid}` (full detail with all linked source records, confidence evidence, and current activity status). |
| `activity.py` | Handles activity status queries. Most importantly: `GET /api/activity/query` — **the demo query endpoint** that accepts filters (status, pin code, days since last inspection) and returns matching UBIDs. Also: `GET /api/activity/{ubid}/timeline` (all events chronologically). |
| `review.py` | Handles the human reviewer workflow. Endpoints: `GET /api/review/queue` (paginated task list), `GET /api/review/task/{id}` (full review card with both records, feature scores, SHAP values), `POST /api/review/task/{id}/decision` (submit CONFIRM_MATCH / NON_MATCH / PARTIAL / REQUEST_INFO / DEFER), `GET /api/review/stats` (auto-link rate, queue depth, override rate). |
| `admin.py` | Admin-only endpoints. `GET /api/admin/audit-log` (full append-only decision trail, RTI-compatible), `GET /api/admin/model-stats` (current model version, override rate, last retrain date), `POST /api/admin/thresholds` (update auto-link and review threshold values without redeploying). |

| `activity_schemas.py` | Models for activity query and timeline endpoints. Defines `ActivityQueryParams`, `ActivityQueryResult`, `ActivityEvent`, `EvidenceSummary`. |
| `review_schemas.py` | Models for the reviewer queue. Defines `ReviewTask`, `ReviewTaskDetail`, `ReviewCard`, `DecisionRequest`, `ReviewStats`. |
| `admin_schemas.py` | Models for admin endpoints. Defines `AuditLogEntry`, `ModelStats`, `ThresholdUpdateRequest`. |

### Phase Wise Planning
Phase 1 — Synthetic Data: Not just "generate 5,000 entities" — the plan gives you the exact Python dataclass schemas for every entity field, the ENTITY_TYPE_DISTRIBUTION dict with real percentages, PAN_PRESENCE_RATE per department, the VARIATION_CONFIG dict with 4 variation categories (legal suffix, abbreviation, typo, transliteration) each with probability weights, the exact EVENT_PATTERNS dict per business status (active/dormant/closed/seasonal), and the specific 12-month date range to use.


Phase 2 — Database: Full SQLAlchemy ORM for all 11 tables with correct types, indexes, foreign keys, JSONB columns, and array columns. Not pseudocode — actual importable Python.


Phase 3 — Normalisation: Complete working implementations of all 5 modules — the full regex list for legal suffixes, the CITY_NORMALISATION dict, the Kannada transliteration pipeline, the Karnataka-specific address parser handling all 5 address format types (BBMP, industrial, survey, landmark, minimal), and the deterministic PII scrambler with HMAC-SHA256.



Phase 4 — ER Pipeline: Full blocker.py with all 6 blocking keys including H3 geo-cell logic, complete feature_extractor.py with all 13 features and the exact -1/0/0.5/1.0 PAN scoring schema, and scorer.py with the LightGBM + Platt Scaling + SHAP pipeline including the TreeExplainer call pattern.



Phase 5 — Activity Engine 

Standalone Python module. Reads from activity_events table; writes to activity_scores.

- *event_router.py*
  - Poll unprocessed events from activity_events (check processed = false)
  - Look up source record ID in ubid_source_links
  - If found → attach UBID, write to ubid_activity_log
  - If not found → write to unmatched_events with triage reason code

- *signal_scorer.py*
  - Define signal weight + half-life constants in a config dict per event type
  - For each UBID, fetch all events in *12-month lookback window*
  - Apply decay formula: score += weight × exp(−λ × days_since_event) where λ = ln(2) / half_life
  - Normalise via sigmoid transform → Activity Score (AS) in [−1, +1]
  - Key weights: Electricity ≥50% baseline = +0.90, Licence renewal = +0.80, Closure = −1.00 (permanent)

- *activity_classifier.py*
  - AS > 0.4 → *Active*
  - −0.2 ≤ AS ≤ 0.4 → *Dormant*
  - AS < −0.2 → *Closed (Suspected)*
  - Hard closure event present → *Closed (Confirmed)* — overrides all positive signals
  - Build evidence snapshot: list of contributing events with individual weighted scores
  - Write classification + JSONB evidence to activity_scores




Phase 6-7 — API + Frontend: All router files with actual endpoint implementations, the complete ReviewPage.jsx with the SHAP waterfall chart using recharts, and the ActivityPage.jsx with the demo query form and timeline modal — things judges will actually see.


Phase 8 — Connecting Everything 

- Frontend calls FastAPI via `axios` — all requests to `http://localhost:8000`
- FastAPI uses SQLAlchemy sessions to read/write PostgreSQL
- ML pipeline scripts are either:
  - Called as FastAPI background tasks for real-time operations, or
  - Run as **scheduled Python scripts** that write pre-computed results to DB (preferred for demo stability)
- Run full pipeline as a **one-shot pre-computation** before demo to populate UBID registry

**Docker Compose (mandatory for demo reliability):**
```yaml
services:
  postgres: ...        # UBID registry + source DB
  nominatim: ...       # self-hosted geocoding
  mlflow: ...          # model tracking dashboard
  backend: ...         # FastAPI + Uvicorn
  frontend: ...        # React dev server or nginx
```
- One `docker compose up` brings the entire system live
- Ensures reproducible demo environment — no "works on my machine" failures



Phase 9 — Demo: A full run_pipeline.py one-shot script and reset_demo.py so you can reset to a clean state 30 minutes before presenting.
